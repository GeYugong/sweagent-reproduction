> `docs/full_paper_reproduction_plan.md`，逐项机器清单为

## 1. 研究问题

本研究围绕四个问题展开：

1. 在论文时期的代码与 ACI 配置下，SWE-agent 能否在小规模 SWE-bench Lite 子集上重现论文所报告的行为模式？
2. ACI 的编辑、搜索、文件窗口和上下文管理组件分别如何影响成功率、成本、时延与失败类型？
3. 在相同任务和预算下，论文版代理、现代代理实现、API 模型和本地代码模型之间存在什么性能与成本差异？
4. 能否从作者原始工件与严格原模型重跑两条证据链，重新生成论文全部经验表格、图形和分析结论？

## 2. 成功标准

### 最低成功线

- 完成一个实例的端到端推理、补丁生成和 Docker 化判分。
- 保存完整轨迹、模型请求统计、补丁、测试输出和判分结果。
- 相同代码与配置能够从实例粒度重放。

### 标准成功线

- 冻结 20 个开发实例并完成论文版基线。
- 对比默认 ACI 与一个弱化 ACI，验证结果方向与论文结论一致。
- 报告 resolve rate、配对结果、95% bootstrap 置信区间、时延、token 和成本。

### 进阶成功线（不等于全论文完成）

- 在 SWE-bench Lite 正式集上完成固定配置评测。
- 加入现代代理和本地模型对照。
- 完成一个单因素改进及对应消融与失败分析。

### 完整成功线

- 完成总方案 E01–E18、B01 和 A01–A14 的全部必需项；
- 从作者原始工件重新计算论文全部经验表图；
- 使用论文精确模型重跑主结果、消融、超参数、pass@k、Full 和 HumanEvalFix；
- 通过干净 clone 自动重建最终表图和证据索引。

## 3. 阶段划分

### P0：协议与版本冻结

- 固定论文 PDF、源码和代码提交。
- 提取论文主结果、提示词、工具接口与超参数。
- 审计本地和服务器的 GPU、CPU、内存、磁盘、Python、Git 与容器运行时。
- 建立 Git 仓库、实验台账、数据清单和输出约定。

退出条件：所有版本选择和环境限制均有书面记录。

### P1：单实例闭环

- 确认 Docker 或经验证的兼容容器后端。
- 安装隔离的项目环境，不修改共享环境。
- 从 SWE-bench Lite dev 中选择一个安装成本较低的实例。
- 完成 inference、patch 保存、evaluation 和 trajectory inspection。

退出条件：单实例能够从干净状态完整重放，且失败时有明确分类。

### P2：冻结开发集

- 依据仓库、年代、补丁规模和安装成本分层选择 20 个实例。
- 在查看正式结果前冻结 `data/manifests/dev20.jsonl`。
- 运行论文版默认配置和一个 shell-only/弱化 ACI 对照。

最初冻结的 `no_search_editor` 同时移除了 `search.sh` 与 `edit_linting.sh`。
完整论文审计确认它既不是论文 Shell-only，也不是论文表中的单因素消融，故已
改列为探索性自定义组合消融。论文对齐实验改为分别运行 no-lint editor、
no-edit、iterative-search、no-search、window-30、full-file、full-history 和
no-demo；Shell-only 作为独立系统恢复。

退出条件：结果表中不存在缺失的运行元数据，失败实例均有分类。

### P3：论文版部分复现

- 固定模型、temperature、文件窗口、历史管理和最大成本。
- 至少执行一个重复批次，估计代理随机性。
- 使用 McNemar 检验比较配对成功率，使用 bootstrap 计算置信区间。

### P4：现代与本地模型对照

- 现代实现和论文版使用相同实例清单与评价协议。
- API 与本地模型采用可比的上下文和预算上限。
- 本地模型先完成单卡推理校准，再评估是否需要双卡并行。

### P5：单因素改进

候选方向按优先级排列：

1. 失败后结构化恢复提示；
2. 工具输出的证据保留与压缩；
3. patch 提交前的强制测试检查。

每次只改变一个因素，不在正式集上搜索超参数。

### P6–P9：全论文矩阵

- P6：论文原模型的 Full、Lite、HumanEvalFix 与六次重复；
- P7：失败模式、轨迹、patch、定位和全部附录分析；
- P8：严格分离的现代模型复验；
- P9：干净环境重放、覆盖率审计和最终报告。

各阶段的 episode 数、外部门槛与退出条件见完整总方案。

## 4. 指标与统计

每个实例至少记录：

- `resolved`、退出状态和判分状态；
- 输入、输出、缓存与总 token；
- 模型调用次数、工具调用次数和交互轮数；
- 推理时长、环境构建时长和评测时长；
- patch 文件数、增删行数和测试变化；
- 失败类型与人工复核备注。

同一实例上的二元成败比较使用 McNemar 检验。连续指标使用配对分析并同时给出稳健的 bootstrap 95% 置信区间。样本量较小时不以单一 p 值作为结论依据。

## 5. 资源纪律

- 默认只使用两张 A40，不因节点可见更多 GPU 而扩大资源占用。
- 正式批量运行前先报告磁盘增量估算。
- Docker 镜像、数据缓存和模型权重不得进入 Git。
- 任何 API 批量调用前先完成一次调用的 token 与费用测量，并设置硬预算。
- 服务器项目目录固定为 `/public/home/mty/GeYugong/05_sweagent_repro_ser/`。
